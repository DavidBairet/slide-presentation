Annuaire de tatoueurs

Application web de recherche, publication et modération de profils d'artistes tatoueurs.

Bonjour, je vais vous présenter mon projet d'oral DWWM : un annuaire de tatoueurs. L'objectif est de montrer une application web complète, depuis l'analyse des besoins jusqu'au déploiement sur un VPS, c'est-à-dire un Virtual Private Server. Je vais suivre l'ordre de mon dossier projet : présentation, analyse, environnement technique, réalisations Front-End, réalisations Back-End, sécurité, déploiement, difficultés, résultats, conclusion et évolutions futures.

1. Présentation du projet

Contexte

Un outil pensé pour rapprocher visiteurs et artistes tatoueurs.

Recherche locale Styles Profils Galeries
Le projet part d'un besoin simple : faciliter la mise en relation entre des personnes qui cherchent un tatoueur et des artistes qui veulent présenter leur travail. L'authentification permet d'identifier les utilisateurs. Une règle importante est de ne jamais stocker un mot de passe en clair. La gestion des rôles permet de différencier les droits. Un visiteur peut consulter. Un artiste peut proposer ou gérer son profil. Un administrateur peut accéder à l'espace de modération. Ce middleware vérifie si l'utilisateur stocké en session possède le rôle admin. Si oui, la requête continue avec next. Sinon, l'utilisateur est redirigé vers la page de connexion administrateur.

5. Réalisations Back-End

Recherche multicritères

const filter = { status: "approved" };

if (q) {
  const qNorm = normalizeForSearch(q);
  const tokens = qNorm.split(" ").filter(Boolean).slice(0, 8);

  const andSearchText = tokens.map((token) => ({
    searchText: { $regex: escapeRegex(token), $options: "i" },
  }));

  filter.$or = [
    { $and: andSearchText },
    { name: rxContains },
    { city: rxContains },
    { styles: { $elemMatch: { $regex: rxContains } } },
    { postalCode: rxPrefix },
    { department: rxPrefix },
  ];
}

const artists = await Artist.find(filter)
  .sort({ createdAt: -1 })
  .skip(skip)
  .limit(limit);
La recherche multicritères filtre d'abord les artistes approuvés. Ensuite, si l'utilisateur saisit une recherche, je normalise le texte pour limiter les différences liées aux accents ou à la casse. Les tokens permettent de rechercher plusieurs mots. Le filtre teste ensuite plusieurs champs : nom, ville, styles, code postal ou département. La pagination utilise skip et limit afin de ne pas charger toute la base d'un coup.

5. Réalisations Back-End

Pagination

const pageParam = parseInt(req.query.page || "1", 10);
const page = Math.max(Number.isFinite(pageParam) ? pageParam : 1, 1);
const limit = 12;
const skip = (page - 1) * limit;

const artists = await Artist.find(filter)
  .sort({ createdAt: -1 })
  .skip(skip)
  .limit(limit);
La pagination évite d'afficher tous les artistes d'un coup. On lit le numéro de page dans la requête, on le sécurise avec Math.max pour éviter une valeur inférieure à 1, puis on fixe une limite de 12 artistes par page. La variable skip permet de sauter les résultats des pages précédentes. Cette logique améliore la lisibilité pour l'utilisateur et la performance côté serveur, car la requête charge seulement une partie des résultats.

6. Sécurité et veille technologique

Protection CSRF

Avant

Formulaire sensible sans token.

Après

Formulaire avec token CSRF vérifié côté serveur.

CSRF signifie Cross-Site Request Forgery. C'est une attaque où un site malveillant peut essayer de déclencher une action à l'insu d'un utilisateur déjà connecté. Par exemple, si l'utilisateur est connecté à l'administration, une requête non voulue pourrait tenter de modifier une donnée. Le token CSRF ajoute une valeur unique dans le formulaire. Côté serveur, on vérifie ce token avant d'accepter l'action. Si le token est absent ou invalide, la requête est refusée.

Secrets

La sécurité est abordée à plusieurs niveaux. Les secrets ne doivent pas être versionnés. Les variables d'environnement permettent de séparer le code et la configuration. MONGODB_URI contient l'adresse de connexion à MongoDB Atlas. SESSION_SECRET sert à signer les sessions. ADMIN_EMAIL peut servir à identifier ou créer un compte administrateur. BASE_URL indique l'adresse publique du projet. Les sessions protègent les espaces connectés. Les mots de passe sont hachés. Les accès administrateur sont contrôlés. En production, SSL/TLS correspond à la sécurisation HTTPS : les échanges entre le navigateur et le serveur sont chiffrés. J'ai aussi gardé en tête les recommandations OWASP, Open Web Application Security Project, notamment sur l'authentification, les injections et la protection des données sensibles. CSRF signifie Cross-Site Request Forgery : c'est une attaque qui pousse un utilisateur connecté à exécuter une action sans le vouloir; elle doit être prise en compte pour les formulaires sensibles.

7. Déploiement du projet

Production

Le déploiement consiste à rendre le projet accessible en ligne. Le code est versionné avec Git et GitHub, puis déployé sur un VPS Linux. VPS signifie Virtual Private Server. L'accès au serveur se fait avec SSH, Secure Shell. Le nom de domaine passe par le DNS, Domain Name System, pour pointer vers le serveur. Enfin, SSL/TLS sécurise l'accès HTTPS. Le serveur héberge l'application Node.js, tandis que MongoDB Atlas héberge la base de données.

7. Déploiement du projet

Sécurisation serveur VPS

ssh david@adresse-du-serveur
sudo ufw status
sudo fail2ban-client status
Un VPS est un Virtual Private Server, c'est-à-dire un serveur privé virtuel. Pour l'administrer, on utilise SSH, Secure Shell, qui permet une connexion distante sécurisée. UFW est le pare-feu : il sert à contrôler les ports ouverts, par exemple HTTP, HTTPS et SSH. Fail2Ban surveille les tentatives répétées de connexion et peut bloquer temporairement des adresses IP suspectes. Cette partie montre que le déploiement ne se limite pas à copier le code : il faut aussi penser à la sécurité du serveur.
